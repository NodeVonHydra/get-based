Architecture Roadmap ​

Current State (v1.12.x) ​

35 vanilla ES modules (~25K lines), no build tools, no framework. All data in localStorage/IndexedDB per browser. AES-256-GCM encryption at rest. PWA with service worker. CORS solved via Vercel Edge proxy (api/proxy.js). Cross-device sync via Evolu CRDT (E2E encrypted, self-hosted relay at sync.getbased.health). OpenClaw context gateway on relay port 4001. Current repo: elkimek/get-based.

What's Been Shipped ​

The original plan assumed sync would force a build step, triggering the TS/Svelte chain. Instead, sync and OpenClaw shipped pragmatically on top of vanilla JS — proving the features without the rewrite.

Phase 0: OpenClaw Context Gateway ✅ ​

  • Context gateway on relay (:4001) pushes buildLabContext() to OpenClaw
  • Read-only — OpenClaw reads health data, users discuss in Signal/Telegram
  • Shipped as part of relay infrastructure, no MCP server needed

Phase 1: Cross-Device Sync ✅ ​

  • Evolu CRDT sync live (v1.9.5–1.9.8), self-hosted relay on VPS
  • BIP-39 mnemonic identity, E2E encrypted
  • Setup wizard (new/join), all profiles + AI settings synced
  • Data model unchanged — importedData serialized as blobs to Evolu (not normalized rows)
  • No build step introduced — Evolu vendored in vendor/

What differs from the original plan: sync was bolted onto vanilla JS rather than built into a typed shared package. The data model redesign (nested arrays → normalized SQLite rows) is deferred to the v2 rewrite.

v2 Migration (Remaining Phases) ​

New Repo, Not a Branch ​

The v2 migration goes into a new repo (get-based-v2 or similar), not a branch on get-based. Reasons:

  • Monorepo structure is fundamentally different from the current flat layout
  • Build tooling (Vite, Svelte plugin, tsconfig) would pollute the current repo's "no build tools" identity
  • Current repo keeps working during migration — users, contributors, and daily driver unaffected
  • Bug fixes cherry-picked from get-based → get-based-v2 during transition
  • Clean git history — no 500-commit migration branch

When v2 is ready: archive get-based, rename get-based-v2 → get-based.

Phase 2: Project Scaffold + Shared Layer (3-5 sessions) ​

New repo. Extract logic from vanilla JS into typed packages.

  • 2a: Project scaffold — monorepo structure, Vite, tsconfig, SvelteKit skeleton
  • 2b: Extract shared layer — schema, constants, data pipeline, API layer → packages/shared as TypeScript. Pure logic, no DOM — safest to migrate first
  • 2c: Evolu data model redesign — importedData nested object → normalized SQLite rows (profile_id, category, marker, date, value). Migration path from v1 blob format. This is the biggest technical risk. The normalized model also unblocks two open issues that are structural limitations of v1's blob/array model:
    • #39 (keep PDF imports separate): v1 merges all markers by date — no provenance tracking. Normalized rows get a source column (filename, import timestamp), enabling selective deletion per import
    • #57 (move markers between categories): v1 marker keys embed the category (biochemistry.glucose), so moving a marker breaks all data references. Normalized model stores category as a column on the marker definition — reassignment is a simple update without breaking value history
  • 2d: Verify round-trip — import data, sync across devices, prove parity with v1

Phase 3: Svelte UI (4-5 sessions) ​

Module-by-module migration using DevTools MCP for visual QA.

Migration order (simple → complex):

  1. Context cards, notes, supplements — repetitive patterns, isolated
  2. Settings, export, glossary, nav — modal-based, self-contained
  3. Charts, views, detail modal — Chart.js integration, plugin lifecycle
  4. Chat, pdf-import — streaming, progress bars, imperative DOM
  5. Tour, changelog, feedback — small, low-risk

Workflow per module:

  • Migrate code
  • Subagent takes before/after screenshots (keeps main context clean)
  • Subagent clicks through key flows, reports pass/fail
  • Commit
  • User does quick sanity check

Context window strategy:

  • Screenshots go to subagents, never the main conversation
  • Convention doc (MIGRATION.md) established in session 1, read at start of each subsequent session
  • Git commits as checkpoints — if session breaks, read git log + existing code to rebuild context

Framework choice: Svelte

  • Best DX for solo maintainer (bind shortcuts, reactive stores, less boilerplate)
  • SvelteKit for web app (routing, static build, SSR if needed later)
  • First-class Tauri support
  • Compile-time reactivity (no virtual DOM overhead — matters for Chart.js-heavy rendering)

Phase 4: SvelteKit Web + Tauri Desktop (3-5 sessions) ​

  • SvelteKit static PWA replaces vanilla JS web version
  • Tauri wraps same Svelte UI for desktop
  • GitHub Actions CI for Mac/Windows/Linux builds
  • Auto-updates via Tauri's built-in updater

Phase 5: Read/Write OpenClaw (2-3 sessions) ​

  • Read/write: user drops lab PDF in Signal/Telegram, OpenClaw runs import pipeline, saves to Evolu sync store, replies with summary
  • Upgrades Phase 0's read-only context gateway
  • OB1/Open Brain as optional cross-tool knowledge layer

Monorepo Structure ​

packages/
  shared/     — TS types, schema, constants, data pipeline, Evolu sync
  ui/         — Svelte components (charts, modals, cards, chat)
  web/        — SvelteKit static PWA
  desktop/    — Tauri (Rust backend + same Svelte UI)

Tauri Desktop Benefits ​

  • Native keychain for API keys (not localStorage)
  • Real filesystem backup (no File System Access API permission dance)
  • Direct API calls (no CORS, no proxy)
  • Background sync
  • Taskbar/dock presence, auto-updates, no browser chrome

v1 Codebase Audit (March 2026) ​

Comprehensive audit to inform migration scope.

Surface area ​

  • 35 JS modules (~25K lines), 3,592-line monolithic CSS, 199-line HTML shell
  • 28 browser-based test files (~7,500 lines)
  • 643 functions on window.* across 25 modules (the #1 migration obstacle)
  • 359 inline onclick= handlers in template-literal HTML
  • 630 template-literal HTML blocks → become Svelte templates
  • 326 scattered localStorage accesses across 21 modules (no abstraction layer)
  • 1 circular dependency (profile.js ↔ crypto.js), 1 duplicated function (_getCyclePhase)
  • 4 independent modal/overlay implementations → unify into <Modal> component

What migrates easily ​

  • Pure data/logic modules (~3,200 lines: state, utils, schema, constants, adapters, hardware, image-utils, food-contaminants, supplement-warnings) — zero DOM, trivial .js → .ts
  • CSS design tokens (33 vars, clean dark/light) — preserve as global theme.css
  • CSS prefix convention (chat-*, cl-*, emf-*, ctx-*) already maps to component boundaries
  • Vendor deps: Chart.js, pdf.js, fonts → npm. Evolu needs WASM/COOP config spike

Test gaps (verify migration parity) ​

No dedicated tests for: views.js (2K lines), data.js, settings.js, context-cards.js, export.js, nav.js, profile.js, adapters.js

Issues solved by v2 data model ​

  • #39 — per-import provenance and selective deletion (blocked by v1's date-merge model)
  • #57 — marker category reassignment without breaking history (blocked by v1's category.marker key format)

When to Start ​

v1 vanilla JS app is the working prototype and daily driver. Every feature shipped now teaches what the real data flow should be. Start v2 when blocked by the architecture, not when bored with it.
